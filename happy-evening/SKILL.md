# Happy Evening Skill

Закрытие рабочей сессии: итоги дня, актуализация Wiki, срез на завтра.

## Параметры

- Аргумент: нет.

## Константы

- **Пользователь Wiki**: `users/g.deshin`
- **Lead (ответственный)**: `g.deshin`
- **Slug страницы недели**: `users/g.deshin/plany-rabot-po-nedeljam/YYYY-WNN`
- **State-файл**: `~/.claude/projects/-home-gd-projects-gd-yandex-tracker-mcp/skill-state.json`
- **Почтовые папки**: `["INBOX", "Outbox", "Sent", "tracker"]`

## State-файл

Общий с `good-morning` и `daily-digest`. Читать в начале, записать `last_mail_read` и `last_happy_evening` в конце.

## Архитектура: Pipeline с обогащением контекста

Те же принципы, что и `/good-morning`:
- Каждый шаг информирует следующий
- Содержательная информация, не метаданные
- Максимум 2 параллельных MCP-вызова (почта + календарь). Tracker API — последовательно
- Все мутирующие действия — только после подтверждения пользователя
- Проект = один блок, группировка по клиенту (не по технологии)
- @mention = FYI (действие только при прямом запросе в теле комментария)

**Особенность вечернего навыка:** почта читается за ВЕСЬ ДЕНЬ (с 00:00), не с `last_mail_read`. Это нужно для полной картины итогов. Чтение попроектно с анализом в основном потоке между запросами.

## Алгоритм

### Шаг 0 — Подготовка

1. Определить текущую дату, день недели, ISO-неделю.
2. Прочитать `skill-state.json`.
3. Вычислить: `today_start` = сегодня 00:00 MSK, `tomorrow` = завтра 00:00 MSK.
4. Определить: пятница ли сегодня?
5. **Проверить локальные накопительные файлы** (`ls tmp/` в рабочем каталоге проекта): матрицы, копилки, черновики (`matrix`, `dopolnenia`, `kp_*`, `ITECO__*`, `draft*`). Учесть в итогах дня: что создано/обновлено сегодня, что требует продолжения на следующий рабочий день.

### Шаг 1 — Почта за день + Календарь завтра (параллельно, разные API)

| # | Вызов | API |
|---|-------|-----|
| 1 | `get_mail_summary(since=<today_start>, before=<tomorrow>, folders=["INBOX","Outbox","Sent","tracker"])` | IMAP |
| 2 | `list_events(start=<завтра>T00:00:00+03:00, end=<завтра>T23:59:59+03:00)` | CalDAV |

#### Анализ после Шага 1

Из почты за день:
- **Группировка по проектам/клиентам**: какие проекты были активны сегодня
- **Хронология по каждому проекту**: что пришло → что ответили → что осталось открытым
- **Tracker-нотификации**: какие задачи изменились за день

Из календаря завтра:
- Встречи на завтра (или на понедельник, если сегодня пятница)
- Связь с проектами

### Шаг 2 — Недельный план (1 запрос к Wiki API)

```
get_wiki_page(slug=<текущая_неделя>)
```

→ Сопоставить план с реальностью: что из Q1 закрыто? Лягушка дня съедена? Кусочек слона сделан?

### Шаг 3 — Глубокое чтение (последовательно, попроектно)

Для каждого проекта с активностью за день (из Шага 1), по убыванию приоритета:

1. Прочитать ключевые письма дня (`get_mail`)
2. Если есть tracker-нотификации — прочитать комментарии (`get_comments`)
3. Прочитать Wiki-страницу проекта (`get_wiki_page`) — для сверки и подготовки обновления
4. **АНАЛИЗ**: что изменилось за день? Что обновить на Wiki?

**Между проектами** — естественная пауза для анализа. Не перегружать API.

### Шаг 4 — Итоги дня в чат

Вывести содержательное саммари:

```
Итоги дня — среда 15.04, неделя W16.

Сделано сегодня:
  - Инфопарк: ОТКАЗ Polymedia, но IPMatika ищет партнёров (тендер до 17.04)
  - Ecoshipping: ОТКАЗ, проект закрыт, резерв IPMatika снят
  - УК Первая: панели в пути (накладная 497-018934289), ожидаем сегодня
  - Mondelez: NDA подписан в ЭДО, ждём проектную документацию
  - Новаметрикс: встреча прошла, бюджет ~640 млн, 2 вопроса КРОКу отправлены

Не сделано (перенос):
  - ГТЛК КП — ждём ответ Акимова, если нет до завтра → бюджетная оценка
  - ТехноНИКОЛЬ — звонок Синтикову не сделан

Лягушка дня (Ср): Закрыть отказы — ✅ Сделано
Слон (Норникель): принять статус Храмова — ✅ Ознакомился

На завтра (Чт 16.04):
  Лягушка: DSOL/Комус — написать письмо
  Слон: Норникель — подготовить данные Северной
  Q1: проверить доставку панелей УК Первая, ГТЛК КП если нет ответа
  Календарь: нет встреч
  Напоминание: УК Первая — продление договора 230525-01
```

**Особый случай — рабочие выходные (Пт вечер с аргументом `weekend=расширить`):**

Если пользователь отметил, что выходные рабочие (аргумент `/happy-evening weekend=расширить`, `no-monday` и т.п.), выводить план детально по Сб и Вс вместо «на завтра»:

```
### План на выходные (рабочие)

Сб DD.MM — приоритеты:
  🔥 [Критичное обещанное, дедлайн] — <задача>
  🔥 [Операционка] — <задача>
  🟡 [Тактика] — <задача>

Вс DD.MM — приоритеты:
  🔥 [Стратегия/углубление] — <задача>
  🟡 [Черновики для понедельника] — <задача>

Календарь выходных: <встречи или «пусто, чистое время»>

На Пн DD.MM — только напоминания (план не делаем, создаём утром через /good-morning):
  - <пинг 1>
  - <пинг 2>
```

Принцип разделения по дням: **Сб — операционный бэклог** (обещанные клиентам артефакты, заполнение документов, отправка обещанного), **Вс — стратегический бэклог** (черновики, углублённая работа, подготовка к понедельнику).

Показать пользователю ТОЛЬКО то, что обновляется. Формат:

Дождаться подтверждения. Выполнить обновления только после "да".

### Шаг 6 — Обновить Wiki (после подтверждения)

По каждому проекту с изменениями:
```
update_wiki_section(slug, heading="Хронология", mode="append", content=...)
```

Обновить недельный план:
- Лягушки: статусы
- Слоны: прогресс
- Закрытые задачи: `- [ ]` → `- [x]`
- Обновить `*Обновлено: ...*`

### Шаг 7 — Пятничная логика

Если сегодня пятница:

```
Сегодня пятница! Создать план на следующую неделю (W17)?
```

Если "да" → предложить запустить `/good-morning` с аргументом следующей недели.

### Шаг 8 — Обновить state-файл

```json

  "last_happy_evening": "<текущее_время_ISO>"

```

Сохранить остальные ключи без изменений.

## Важные правила

### Архитектура
- **Почта за весь день** (с 00:00), не с `last_mail_read` — для полной картины итогов
- **Чтение попроектно** с анализом в основном потоке — не перегружать API
- **Rate limits**: максимум 2 параллельных MCP-вызова (почта + календарь). Tracker API — последовательно
- **Глубина контента**: читать письма, комментарии — давать саммари, не ID/счётчики

### Классификация и группировка
- **Проект = один блок**: группировка по клиенту, не по технологии
- **Партнёр ≠ клиент**: КРОК — партнёр, Новаметрикс — клиент
- **@mention = FYI**: действие только при прямом запросе в теле комментария
- **Просрочки и отказы**: в начало итогов, как и в `/good-morning`

### Действия и подтверждения
- **Все мутирующие действия — только после подтверждения**
- **Черновик обновлений Wiki в чат** — показать ТОЛЬКО то, что меняется
- **Wiki проектов обновляется дважды в день**: утром (`/good-morning`) и вечером (`/happy-evening`)

### Вывод
- **Итоги**: сделано / не сделано / перенос — по проектам
- **Лягушка/слон**: статус за день
- **Срез на завтра**: лягушка + слон + Q1 + календарь + напоминания
- **Пятница**: предложить создать план на следующую неделю
- **Не упоминать локальные пути** в Wiki
- **Финансовые данные запрещены** в Tracker и Wiki-планах
