# Good Morning Skill

Утреннее открытие рабочей сессии. Pipeline с нарастанием контекста: почта → Wiki → задачи → классификация → план.

## Параметры

- Аргумент (опционально): номер ISO-недели. По умолчанию — текущая неделя.
- Аргумент (опционально): `planerka` — принудительно сформировать брифинг для планёрки (Шаг 8.6) независимо от дня недели. По умолчанию брифинг формируется автоматически только в понедельник.

## Константы

- **Пользователь Wiki**: `users/g.deshin`
- **Lead (ответственный)**: `g.deshin`
- **Slug индекса**: `users/g.deshin/plany-rabot-po-nedeljam`
- **Slug страницы недели**: `users/g.deshin/plany-rabot-po-nedeljam/YYYY-WNN` (например `2026-W16`)
- **State-файл**: `/Мерусофт/state/skill-state.json` (Яндекс Диск)
- **Почтовые папки**: `["INBOX", "Outbox", "Sent", "tracker"]`

## State-файл

Все три навыка (`good-morning`, `daily-digest`, `happy-evening`) разделяют один файл состояния на Яндекс Диске:

```json
{
  "last_mail_read": "2026-04-15T16:37:00+03:00",
  "last_good_morning": "2026-04-15T08:30:00+03:00",
  "last_digest": "2026-04-15T15:00:00+03:00",
  "last_happy_evening": "2026-04-14T18:00:00+03:00"
}
```

**Чтение** (начало навыка):
```
download_disk_file(path="/Мерусофт/state/skill-state.json")
```
Если файл не найден — считать все timestamps = `null` (первый запуск).

**Запись** (конец навыка):
```
upload_to_disk(path="/Мерусофт/state/skill-state.json", content=<json>, overwrite=true)
```
Сохранять все ключи, обновляя только нужные.

- `last_mail_read` используется как `since` для `get_mail_summary`. При `null` — брать пятницу прошлой недели.

## Архитектура: Pipeline с обогащением контекста

**Принцип:** каждый шаг информирует следующий. Никогда не запрашивать данные вслепую. Максимум 2 параллельных MCP-вызова в одном блоке (почта и календарь используют разные API и не конфликтуют). Все запросы к Tracker API — последовательно с анализом между ними.

**Принцип глубины:** навык должен давать СОДЕРЖАТЕЛЬНУЮ информацию, а не метаданные. Не "Bayer: 4 письма", а "Bayer: Zhelyadinova пишет что ДС №17 согласовано, нужна подпись до пятницы". Не "MIMP-891: Норникель [Автор]", а "Храмов просит проверить подложку 11 этажа Невы, приложил скриншот с ошибкой в зонировании".

## Алгоритм

### Шаг 0 — Подготовка

1. Определить текущую ISO-неделю, даты: понедельник, воскресенье, сегодня. Определить **день недели**: если сегодня понедельник (или навык вызван с аргументом `planerka`) — на Шаге 8.6 формируется брифинг для планёрки.
2. Скачать state с Диска: `download_disk_file(path="/Мерусофт/state/skill-state.json")`. Если не найден — первый запуск.
3. Вычислить `since` для почты: `last_mail_read` из state-файла, или пятница прошлой недели если `null`.
4. Определить `tomorrow` = завтра, для верхней границы почтового запроса (`before`).
5. **Проверить локальные накопительные файлы** предыдущих сессий:
   - `ls tmp/` в рабочем каталоге проекта — получить список .md/.xlsx черновиков
   - Имена с ключевыми словами: `matrix`, `dopolnenia`, `kp_*`, `ITECO__*`, `черновик*`, `draft*` — показать пользователю в утреннем саммари если актуально (например, незавершённая матрица соответствия ТЗ, копилка допущений КП)
   - Это помогает продолжить работу с того места, где остановились вчера, без потери контекста

### Шаг 1 — Почта + Календарь (параллельно, разные API)

Два параллельных вызова (безопасно — IMAP и CalDAV, не Tracker API):

| # | Вызов | API |
|---|-------|-----|
| 1 | `get_mail_summary(since=<since>, before=<tomorrow>, folders=["INBOX","Outbox","Sent","tracker"])` | IMAP |
| 2 | `list_events(start=<понедельник>T00:00:00+03:00, end=<воскресенье>T23:59:59+03:00)` | CalDAV |

#### Анализ после Шага 1

Из почты извлечь:
- **Группировка по цепочкам** (по теме/контакту): какие проекты/клиенты горячие
- **Из INBOX**: кто пишет, что просит, ждёт ли ответа
- **Из Sent**: на что уже ответил, что отправил
- **Из Outbox**: что написал, но не ушло
- **Из tracker**: какие задачи изменились, кто @упоминает (ПРИОРИТЕТ 0!)
- **Хронология**: сопоставить входящие с исходящими → понять, где цепочка открыта (ждут ответа), где закрыта
- **По каждой цепочке**: Кто? Что? Когда/до какого срока?

> **⛔ Гейт «мяч у нас» (ОБЯЗАТЕЛЬНО перед пометкой «Делай сейчас / нужен ответ»):** `get_mail_summary` отдаёт только ЗАГОЛОВКИ. По каждой открытой цепочке найти САМОЕ ПОЗДНЕЕ письмо по теме (входящее ИЛИ исходящее) и открыть его тело через `get_mail(uid)` — не судить по заголовку:
> - последнее письмо в треде — наш **Sent** (ответ / вложения / оценка) → цепочка **закрыта / в Ожидании**, мяч у клиента, НЕ «Делай сейчас»;
> - последнее — **входящее без нашего ответа** → действительно ждут нас.
>
> Проверять и исходящие РП (Храмов `s.khramov`, Растворцев) — задачу мог закрыть он, а не мы. Пропуск Sent = ложное «срочно» и подрыв доверия к навыку.
>
> **📋 Обязательный артефакт гейта — ВЫВЕСТИ В ЧАТ перед любой классификацией. Пропуск любого из трёх пунктов = недостоверный обзор.**
>
> 1. **Реестр мяча (с UID).** Таблица по каждой открытой цепочке: `Цепочка | Последний UID | Кто прислал | Направление (вх./Sent) | Мяч`. **Правило: пометка «Делай сейчас / мяч у нас» НЕВОЗМОЖНА без UID в этой строке.** Нет UID → тело письма не открыто → метка «мяч у нас» ЗАПРЕЩЕНА. Строка с меткой «мяч у нас» и пустым UID = самопойманный пропуск: вернуться и открыть тело через `get_mail(uid)`.
> 2. **Сверка с памятью проектов.** По строке на каждый активный клиент/проект: `project_X.md — прочитан? → статус в памяти / расхождение`. Если имя клиента ИЛИ контакта (email/ФИО) из почты/Wiki имеет файл `project_*` в auto-memory — файл ОБЯЗАН быть прочитан и процитирован (тот же проект может прийти под другим именем контакта — как контакт Савенко = проект Онегин). Не сверил имя контакта с памятью → развёл один проект на два.
> 3. **Размер дельты НЕ влияет на глубину гейта.** 11 писем и 200 проходят ОДНУ И ТУ ЖЕ процедуру. «Мало писем → неделя тихая → читать нечего» — ЗАПРЕЩЁННАЯ отговорка и корневая причина двух провалов 06.07.2026 (ОЛОВО классифицирован по заголовку «Карточка ООО» вместо тела с финальным пакетом в ЭДО; контакт Савенко не сверён с `project_onegin`).

Из tracker-папки извлечь **номера задач**, по которым пришли уведомления — это входные данные для Шага 4.

Из календаря:
- Встречи на неделю с участниками
- Связь встреч с проектами/задачами

**Результат Шага 1:** список тем/проектов с активностью, номера задач из tracker-нотификаций, расписание встреч.

### Шаг 4 — Глубокое чтение (последовательно, по приоритету)

**Для каждой темы/задачи из merged-списка**, по убыванию приоритета. После каждого блока — анализ в основном потоке, корректировка последующих запросов.

При комбинации ролей (например, задача где пользователь и Исполнитель и @упомянут) — автоматически присваивается **наивысший приоритет** комбинации.

#### 4a. @упоминания из tracker-папки (FYI по умолчанию)

**@mention в Tracker = FYI (for your information).** Призыв означает только "для сведения". Действие требуется ТОЛЬКО если в теле комментария есть прямой текстовый запрос ("проверь", "подтверди", "ответь", прямой вопрос).

По каждой задаче, где пользователя @призвали:

```
manage_comments(operation="list", target={type:"issue", id:issue_key})
```

→ Прочитать комментарий. Есть ли прямой запрос? Если да — это задача. Если нет — принять к сведению. Если нужен контекст переписки:

```
get_mail(message_id) или search_mail(subject=<тема>)
```

→ **АНАЛИЗ**: FYI или прямой запрос? Что конкретно просят (если просят)? Срочно ли?

#### 4b. Приоритет 1: Исполнитель

По каждой задаче, где пользователь — исполнитель (из merged-списка, не все подряд!):

```
get_issue(issue_key)
```

→ Статус, дедлайн, описание. Если есть свежая активность (комментарии в tracker-нотификациях):

```
manage_comments(operation="list", target={type:"issue", id:issue_key})
```

**Если задача связана с проектом** — прочитать Wiki-страницу проекта для полного контекста:

```
get_wiki_page(slug=<slug_проекта>)
```

→ Хронология, коммерция, статусы, решения. Сверить с данными из Tracker и почты. Если информация на Wiki устарела — отметить для обновления на Шаге 8.

→ **АНАЛИЗ**: Что нужно сделать? Какой дедлайн? Есть ли блокеры? Что на Wiki устарело?

#### 4c. Приоритет 2: Автор

По задачам, где пользователь — автор (из merged-списка):

```
get_issue(issue_key)
```

→ Краткий статус: кто исполнитель, что сделано, нужно ли вмешательство.

Если задача связана с проектом — прочитать Wiki-страницу проекта (если ещё не прочитана в 4b).

#### 4d. Приоритет 3: Наблюдатель (только при свежей активности)

Только для задач, по которым пришли tracker-нотификации:

```
get_issue(issue_key)
```

→ Что изменилось? Нужно ли внимание?

#### 4e. Ключевые почтовые цепочки

Для цепочек, где нужно прочитать содержание (определено на Шаге 1 — открытые цепочки, ожидающие ответа):

```
get_mail(message_id)
```

→ Краткое саммари: кто, что просит, нужно ли действие, дедлайн.

**Между каждым блоком** (4a, 4b, 4c, 4d, 4e) — естественная пауза для анализа. Модель оценивает: достаточно ли контекста? Нужно ли уточнить у пользователя? Нужно ли скорректировать приоритет следующих запросов?

### Шаг 6 — Классификация (Ветка A: новый план)

К этому моменту модель имеет **полный контекст** по каждой задаче/теме: не заголовки, а содержание переписок, комментариев, статусов.

#### 6a — Матрица Эйзенхауэра

**Критерии классификации:**

| Фактор | Влияет на "Срочно" | Влияет на "Важно" |
|---|---|---|
| Дедлайн на этой неделе | +++ | |
| Приоритет Tracker: critical/blocker | ++ | ++ |
| Роль: Исполнитель | | +++ |
| Роль: Автор | | ++ |
| Роль: Наблюдатель | | + |
| Связано с активным проектом (in_progress/at_risk) | | ++ |
| Кто-то ждёт ответа (из анализа переписки) | ++ | + |
| Встреча на этой неделе по теме | + | + |
| Перенесено с прошлой недели | ++ | |
| Реальная срочность из содержания писем | Зависит от контента | |

**Матрица:**

| | Срочно | Не срочно |
|---|---|---|
| **Важно** | Q1: Делать первым | Q2: Запланировать |
| **Не важно** | Q3: Делегировать / быстро закрыть | Q4: Отложить / отказаться |

**ВАЖНО:** Классификация на основе ПРОЧИТАННОГО КОНТЕНТА, а не заголовков. Если из переписки видно, что клиент написал "точных сроков нет" — задача НЕ срочная, даже если выглядит срочной по заголовку.

**Правило проекта = один квадрант.** Все задачи/действия одного проекта/клиента получают МАКСИМАЛЬНЫЙ приоритет из комбинации. Если ГТЛК КП (Q1) и ГТЛК on-prem (Q2), то всё ГТЛК → Q1. Пользователь работает проектами, не переключается между квадрантами внутри одного клиента.

**Правило группировки по клиенту, не по технологии.** УК Первая (EINK батарейки) и Ecoshipping (EINK навигация) — РАЗНЫЕ проекты, несмотря на общее слово EINK. Партнёр ≠ клиент: КРОК — партнёр, Новаметрикс — клиент. Проект называется по клиенту.

**Просрочки и отказы → начало Q1.** Задачи с просроченным дедлайном И проекты с отказом клиента автоматически попадают в НАЧАЛО Q1 (отдельный блок "Требует немедленной реакции:"), чтобы пользователь сразу видел что профукано или требует закрытия.

#### 6b — Лягушки недели (7 штук, по одной на день)

Определять ТОЛЬКО после прочтения реального контента.

- Источник: задачи из Q1 и Q2
- Критерии: откладывается (была в прошлой неделе), неприятная/рутинная, требует волевого усилия
- Если подходящих задач меньше 7 — заполнить сколько есть, остальные дни `—`
- Суббота/воскресенье — по умолчанию `—`

#### 6c — Слоны недели (1-3)

- Источник: задачи/проекты из Q2 с большим объёмом
- По каждому: декомпозиция на "кусочки" по дням (1-3ч каждый)
- Если слонов нет — секцию не добавлять

| Дедлайны/приоритеты | Шаг 4 | Изменились → предложить переклассификацию |
| Лягушки | Шаг 4 | Вчерашняя сделана? Сегодняшняя актуальна? |
| Слоны | Шаг 4 | Вчерашний кусочек сделан? Прогресс |
| Календарь | Шаг 1 (events) | Новые/отменённые встречи |
| Почтовый контекст | Шаг 1 (mail) + Шаг 4e | Обновить саммари |

#### Показать изменения в чате (содержательно)

```
### Изменения с последнего обновления (DD.MM HH:MM):

**Закрыты:**
- [x] Норникель Нева 11эт (MIMP-889) — Храмов завершил отрисовку

**Новые задачи (предлагаю добавить):**
- Mondelez — Stepanova прислала NDA, нужна подпись → рекомендую Q2

**Лягушки:**
- Вчера (Пн): ГТЛК КП — Не сделано, перенести?
- Сегодня (Вт): ТехноНИКОЛЬ — позвонить Синтикову

**Слоны:**
- Норникель: вчерашний кусочек "ответить Храмову" — Готово (1/3)
- Сегодня: сверка данных от Богдановой

Подтвердить?
```

Очевидные обновления (закрытые → `[x]`) — без подтверждения.
Переклассификация, новые задачи — с подтверждением.

После подтверждения → `update_wiki_page`, обновить `*Обновлено: ...*`.

### Шаг 8.5 — Актуализация связанных Wiki-страниц (экспресс-схема)

После публикации/обновления недельного плана — обновить связанные Wiki-страницы по четырём категориям: **проекты, продукты, конкуренты, партнёры**. Обновления делать **только при появлении новой содержательной информации** на Шагах 1–4.

#### 8.5a — Проекты

Для каждого проекта, где на Шаге 4 была прочитана Wiki-страница и найдены расхождения:

```
update_wiki_section(slug=<slug_проекта>, section=<секция>, content=<новое_содержание>)
```

**Актуализация идёт со сверкой к эталонному шаблону проектных страниц:**
https://wiki.yandex.ru/users/g.deshin/projects/page-template/

- Шаблон задаёт канонический набор и порядок разделов проектной страницы — сверять структуру обновляемой страницы с ним.
- Если в проектной странице **отсутствует обязательный раздел** шаблона (а данные для него появились на Шагах 1–4) — предложить добавить раздел по шаблону.
- Если структура **расходится** с шаблоном (другие названия/порядок разделов) — отметить в саммари как кандидата на приведение к шаблону (без автоправки структуры — см. «Что НЕ обновлять без подтверждения»).
- Содержимое разделов оформлять в стиле шаблона (заголовки, таблицы, бэктики для брендов/моделей).

**Что обновлять:**
- Хронология: новые письма/встречи/решения
- Статус: текущий статус (presale/in_progress/at_risk/cancelled/closed)
- Контакты: новые участники из переписки
- Коммерция: изменения условий (без финансовых данных!)

#### 8.5b — Продукты (Smart Office и др.)

Обновлять при появлении новых **технических фактов**:
- Новая поддерживаемая модель железа (таблички/планшеты/панели) → в раздел «Оборудование»
- Подтверждённая/реализованная интеграция → в «Интеграции»
- Известные ограничения, выявленные в переписке с заказчиком → в «Ограничения/допущения»
- Конкурентные сравнения (функциональные преимущества/разрывы) → в «Сравнение с рынком»

Источник: письма, комментарии Tracker, материалы тендерных заявок.

#### 8.5c — Конкуренты

Обновлять, если конкурент упомянут в переписке/тендере с новой информацией:
- Новый проект, где заказчик рассматривает конкурента → в «Активные конкуренции»
- Ценовая/функциональная реакция клиента на конкурента → в «Позиционирование»
- Потеря/выигрыш тендера против конкурента → в «История конкуренций»

Триггеры: фразы «рассматриваем UnSpot», «Booco предложил», «в шорт-листе» в переписке.

#### 8.5d — Партнёры (интеграторы/вендоры)

Обновлять при изменениях:
- Коммерческие условия, скидки, партнёрский статус → «Коммерция»
- Активные совместные проекты (presale/реализация/closed) → «Проекты»
- Новые контакты, офис, реквизиты → «Контакты»
- Изменение схемы работы (кто идёт головой, доли выручки) → «Модель сотрудничества»

Триггеры: письма от @croc.ru, @i-teco.ru, @ipmatika.com, @ancomp.ru и т.п.

#### Общие правила

**Что НЕ обновлять без подтверждения:**
- Удаление существующей информации
- Изменение структуры страницы
- Создание новой страницы (для нового партнёра/конкурента/продукта — спросить отдельно)

**Если страница не найдена** — отметить в саммари как «⚠ нет Wiki-страницы для X, создавать?» (без автосоздания).

Все страницы актуализируются дважды в день: утром (`/good-morning`) и вечером (`/happy-evening`).

### Шаг 8.6 — Брифинг для планёрки (только понедельник)

**Условие:** сегодня понедельник (определено на Шаге 0), ИЛИ навык вызван с аргументом `planerka`. В остальные дни шаг пропускается полностью.

**Назначение:** устная шпаргалка к еженедельной планёрке. На планёрке пользователь коротко рассказывает (1) основные результаты прошлой недели и (2) планы на предстоящую неделю, включая встречи. Формат — телеграфный, для проговаривания вслух, НЕ для отправки.

**Источники — всё уже прочитано на Шагах 1–8, новых запросов не делать:**

| Блок брифинга | Источник |
|---|---|
| Результаты прошлой недели | Wiki прошлой недели (Шаг 2): закрытые `- [x]`, съеденные лягушки, прогресс слонов + закрытые задачи Tracker и отправленные письма Sent (Шаги 1, 4) |
| Планы на эту неделю | План текущей недели (Шаг 6/8): главные фокусы Q1/Q2, лягушки, слоны |
| Встречи недели | Календарь недели (Шаг 1, `list_events`) |

**Группировка — по клиенту/проекту** (как во всём навыке), не по технологии. Проект = одна строка.

#### Формат брифинга (в чат)

```markdown
## 🗣 Планёрка — понедельник ДД.ММ

### Результаты прошлой недели
- **Клиент/Проект**: что закрыто/продвинулось — по факту, одной строкой
- ...

### Планы на эту неделю
- **Клиент/Проект**: главный фокус недели — одной строкой
- ...

### Встречи недели
- Пн ЧЧ:ММ — событие (с кем / по чему)
- Ср ЧЧ:ММ — ...
```

**Правила содержания:**
- Каждая строка — телеграфно, как для устного рассказа. Не «MIMP-889 переведён в Closed», а «Норникель — карты 11 этажа сданы».
- Только **содержательные** результаты и планы. Рутину, FYI и watch-задачи не включать.
- Нет результата по проекту за прошлую неделю — не упоминать его в «Результатах» (но он может быть в «Планах»).
- Встречи — все из календаря текущей недели, в хронологическом порядке; если встреч нет — строка «— встреч нет».
- Объём: каждый блок 3–7 строк. Если получается больше — оставить только то, что реально прозвучит на планёрке.

#### Сохранение в Wiki

После показа в чате — записать брифинг как раздел на странице недели текущей недели (создана/обновлена на Шаге 8):

```
update_wiki_section(slug=<текущая_неделя>, section="🗣 Планёрка (понедельник ДД.ММ)", content=<брифинг>)
```

Раздел размещается **вверху** страницы недели, перед Q1 — это входной артефакт недели. Запись — без отдельного подтверждения (брифинг уже показан в чате); финансовые данные не включать.

```
  "last_mail_read": "<текущее_время_ISO>",

```

## Проактивный триггер

При начале любой сессии или вызове `/daily-digest`:

1. Прочитать `skill-state.json`
2. Проверить `last_good_morning`:
   - Если сегодняшняя дата (по MSK) > `last_good_morning` дата **И** текущее время < 12:00 MSK → предложить: *"Доброе утро! Запустить /good-morning?"*
   - Если `last_good_morning` = `null` или дата < сегодня → предложить: *"Сегодня ещё не было утреннего обзора. Запустить /good-morning?"*
3. Не навязывать после 12:00, если пользователь уже запускал что-то другое.

## Важные правила

### Архитектура
- **Pipeline с обогащением**: каждый шаг информирует следующий. Не запрашивать данные вслепую
- **Rate limits**: максимум 2 параллельных MCP-вызова (только почта + календарь — разные API). Всё к Tracker API — последовательно с анализом между запросами
- **Глубина контента**: читать письма, комментарии, описания — давать саммари, а не ID/счётчики
- **Классификация**: на основе ПРОЧИТАННОГО КОНТЕНТА, не заголовков

### Почта
- **4 папки**: INBOX, Outbox, Sent, tracker — всегда
- **Только дельта**: с `last_mail_read`, не перечитывать старое
- **Из почты**: Кто? Что? Когда/До какого срока? + хронология (вх → исх → ответ)

### Классификация Эйзенхауэра
- **Проект = один квадрант**: все задачи одного клиента/проекта получают МАКСИМАЛЬНЫЙ приоритет из комбинации. Не дробить проект по квадрантам
- **Группировка по клиенту, НЕ по технологии**: УК Первая (EINK батарейки) ≠ Ecoshipping (EINK навигация). Партнёр ≠ клиент (КРОК — партнёр, Новаметрикс — клиент)
- **Просрочки и отказы → начало Q1**: отдельный блок "Требует немедленной реакции:" перед основным списком Q1. Отказ клиента = такой же приоритет как просроченный дедлайн (нужно закрыть задачи, уведомить команду)
- **@mention в Tracker = FYI**: призыв означает "для сведения". Действие нужно ТОЛЬКО при прямом текстовом запросе в теле комментария
- **Приоритеты ролей**: Исполнитель (1) > Автор (2) > Наблюдатель (3). При комбинации — наивысший

### Wiki и публикация
- **Никогда не публиковать без подтверждения** при создании нового плана
- **Лёгкие обновления** (закрытые → `[x]`) — без подтверждения
- **Не упоминать локальные пути** в Wiki-страницах
- **Финансовые данные запрещены** в Tracker и Wiki-планах

### Лягушки и слоны
- **Лягушки**: 7 на неделю, определять ПОСЛЕ прочтения реального контента
- **Слоны**: 1-3 на неделю, с декомпозицией по дням

### Планёрка (понедельник)
- **Только понедельник** (или аргумент `planerka`) — Шаг 8.6. В остальные дни не формировать
- **Устная шпаргалка**: телеграфно, для проговаривания вслух, не для отправки
- **Три блока**: результаты прошлой недели + планы на эту + встречи недели
- **По клиенту/проекту**, одной строкой; только содержательное, без рутины и FYI
- **Без новых запросов**: всё из Шагов 1–8; сохранить разделом вверху страницы недели
